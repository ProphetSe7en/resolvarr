Primary vs Secondary — quick primer

Primary is whichever instance a scan or rule is pointed at — the one whose library gets walked, decisions evaluated, and tags written.

Secondary is the mirror target you pick when "sync" is enabled. For each movie tagged on the primary, the same tag is added/removed on the secondary by matching TmdbID. Movies present on the primary but missing on the secondary are simply skipped.

There's no stored "role" — both fields below just describe an instance. Whichever one a given scan/rule targets is its primary for that run; the other instance(s) of the same type are eligible secondaries.

  • Tag release groups (Discover + tag) — coming with M-Sonarr. Sonarr's per-episode-file model needs different aggregation than Radarr's per-movie walk; lands as a future milestone.

What Recover does

Walks every episode file in this Sonarr instance and finds the ones whose releaseGroup field is empty or set to Unknown. For each one, looks up Sonarr's grab history for the parent series, finds the import event that brought this file in, and writes the release group from that grab back onto the file.

Why files end up with missing release groups: indexers sometimes don't include the group in the filename, or the file was hand-imported and Sonarr never had a chance to parse the original release name.

What the result looks like

Each row in the result represents one episode file labelled Series Title — S01E05. A series with 200 missing-RG episodes shows as 200 rows; the per-row exclude checkbox lets you skip individual files when applying.

  • would-fix — Recover found a verified group from grab history; ready to apply.
  • flagged — Filename already carries a group; Sonarr's metadata just doesn't reflect it. Manual verify recommended.
  • no-history / no-rls-group / failed-verify — different states where Recover couldn't determine the group with certainty. Drill in to see why.
Trigger rename after fix: tick this in the result panel before applying to also re-run Sonarr's on-disk RenameFiles command so filenames reflect the recovered group.

Exclude

Some series or seasons just can't be fixed — the indexer dropped the release group and the grab history doesn't help. Rather than seeing them in every scan, click Exclude series on a series header or Exclude season on a season header to skip them in future scans entirely. Saves both time and result-list space.

Excluded items aren't deleted — they live in a Show excluded panel at the bottom of the result. If something changes (the indexer adds the missing data, you re-import the file, etc.), click Include again on its row and the next scan will look at it.

Recover missing release groups
Walks every episode file in Sonarr whose release-group field is empty or Unknown, looks up the matching grab in Sonarr's history, and works out which release group the file came from.

Run Recover is read-only — nothing is written. It opens a result panel showing what could be fixed and why; in there you tick the rows you want, optionally tick Trigger rename after fix, and click Apply selected fixes to commit the changes. Apply is the only step that touches Sonarr.

What is a release group?

The team that encoded or captured a release — names like FLUX, FraMeSToR, BHDStudio, SiC, TheFarm. They appear at the end of a filename after a hyphen: Movie.2024.WEB-DL-FLUX.

Different groups have different reputations. Some only release premium quality (lossless audio, studio-master sources). Others mix premium and standard. Resolvarr lets you tag the ones you want to track.

The three actions on this page

  • Run Discover — walks your library and surfaces release groups that pass your Filters but aren't on the Active list yet. Result pops in a modal — review candidates, tick the ones you want, click Add Selected. Tick Include configured in the actions card for an audit-style view that also shows groups already on Active. Radarr only today.
  • Run Recover — finds movies whose release-group field is empty or Unknown, looks up the matching grab from history, and works out which group the file came from. Read-only — opens a result panel with would-fix / flagged / no-history / etc. buckets and per-row checkboxes; click Apply selected fixes to commit. Per-row Exclude lets you skip rows you can't fix in future scans.
  • Tag release groups — opens a short wizard that walks you through Use active vs Use Discover, filter (mandatory), and Preview vs Apply. Then tags every movie that qualifies. The wizard replaces the legacy stand-alone Tag library card.

Active groups list (below)

Each row is a group you want to track. When the Tag run goes through your library, every movie released by an enabled group on this list gets the matching tag. The mode column controls when the tag is applied:

  • filtered — only tag releases that also pass your Filters rules. Use this for groups that release both premium and standard quality.
  • simple — tag every release from the group, no quality check. Use this for groups where everything they release is premium (e.g. remux groups).

Why tag by release group?

You can use the resulting tags in:

  • Quality Profiles — different upgrade rules per tag
  • Custom Formats — score movies higher/lower based on tag
  • Plex / Jellyfin collections — via Kometa or DAPS
  • Tag-driven cleanup tools — auto-remove movies based on tag (third-party)

A common pattern is a shared premium tag across several groups (FLUX + SiC + TheFarm) so downstream automation just checks one tag.

Preview vs Apply

The Tag release groups wizard's last step (Review) gives you the choice. Preview reads your library and shows exactly which movies would gain or lose tags — nothing is written. Apply commits. Run Preview first if you're not sure; the result panel has an Apply now button to promote the preview to apply without re-running the wizard.

Mirror decisions to a secondary instance

If you have a secondary Radarr (e.g. a 4K instance), tick Mirror tag decisions to secondary on step 1 of the wizard. Resolvarr matches movies by TMDb ID and applies the same tag decisions to the secondary. Orphan cleanup runs automatically when sync is on, so the secondary stays aligned with the primary.

What filters do

These rules decide which releases are "good enough" to get tagged. They apply only to Release Groups in filtered mode — simple groups skip filters entirely and tag every release. Extra tags also ignore filters — they tag straight from media-info regardless of source.

Quality rules

Check where the release came from. MA (Movies Anywhere) and Play (Google Play) WEB-DLs preserve the original studio master audio — unlike AMZN, NF, DSNP, and HMAX, which typically compress to lossy formats. If you only want premium audio sources, leave both on.

Audio rules

Check the audio codec. Lossless formats (TrueHD, DTS:X, DTS-HD MA) preserve every detail of the original mix. Lossy formats (EAC3, AAC, AC3) compress audio to a lower bitrate, which means noticeably lower quality on a good speaker setup.

Releases flagged as upmixed, transcoded, or re-encoded are rejected even if they claim a lossless codec — so your filter doesn't get fooled by a fake.

Cleanup unused tags

The bottom card on this tab finds release-group tags in your with 0 movies attached — leftovers from removed groups, tightened filters, or rejected releases. Lives here because tightening filters is the most common reason tags become orphaned.

Safety: only iterates labels from your Active groups list. Quality-profile, custom-format, and manually-created tags are never touched.

What DV detail tags do

The Video tags HDR bucket emits a single dv tag for any Dolby Vision release — read straight from Radarr's mediaInfo. That's enough for most users. DV detail goes one level deeper and identifies which flavour of Dolby Vision the file is, so custom formats can score them differently.

What gets tagged

  • Profile — dvprofile5 / dvprofile7 / dvprofile8 (which DV encoding the file uses)
  • Layer — fel (Full Enhancement Layer) / mel (Minimal Enhancement Layer)
  • CM version — cm2 / cm4 (the metadata generation embedded in the RPU)
  • No-DV fallback — no-dv when Radarr's mediaInfo flagged the file as DV but extraction couldn't confirm it (no RPU found, or the extraction errored). Lets you write custom formats that prefer files with confirmed DV (real dv + profile/layer tags) over files that claim DV but can't be verified. Untick this in the per-value list if you only want positive detections.

How it works

Radarr's mediaInfo doesn't expose RPU detail (the embedded Dolby Vision metadata layer that carries profile / layer / CM-version) — so DV detail has to read each file directly with ffmpeg + dovi_tool. Both tools ship baked into the image, so DV detail works zero-config out of the box. Results are cached to /config/dv-cache.json so re-scans only touch new or changed files. The Enable toggle below is off by default — turn it on when you want DV-detail tags layered on top of the base dv tag from Video tags.

File access — the one DV-specific gotcha

Reading the file means resolvarr needs to open it from inside its own container. Radarr/Sonarr report a path like /movies/Title.2024.UHD.mkv — that's the path Radarr's container sees. Two scenarios:

  1. Same in-container path in both — if your Radarr template mounts /mnt/user/movies → /movies and you mount the same source as /movies in resolvarr, no extra config is needed. The path Radarr reports works as-is.
  2. Different in-container paths — if Radarr sees /movies but you mounted the share as /media/movies in resolvarr, you need a Path mapping. Go to Settings → Instances → click your Radarr → Path mappings (the section is collapsed by default) and add a row: /movies (Radarr's prefix) → /media/movies (in resolvarr).

If a DV scan result shows Unreachable count > 0, that means os.Stat couldn't open the path — usually scenario #2 unmapped. Open the failed row in the per-movie drill-in to see the exact path resolvarr tried.

Cache — how it speeds things up

Every successful DV extraction is saved to /config/dv-cache.json keyed by movie file ID + size. Re-scans skip files where both match — the ffmpeg + dovi_tool pipeline only runs once per file. Per-file extraction is actually fast on remux sources (tens of milliseconds once dovi_tool finds the RPU SEI in the first GOP), but cumulative + fork/exec/IO overhead across hundreds of files makes the cache worth having.

The size half of the key is the auto-invalidation: when Radarr re-imports an upgrade, the new file has the same movie file ID but a different size — the old cache entry is bypassed and the new file gets re-extracted. No manual refresh needed.

The Clear DV cache button under the Run scan controls forces a full re-extraction on the next scan. Useful when dovi_tool is upgraded (a new version may detect different layer or CM-version semantics on the same file). DV tags already on movies in Radarr are not touched by Clear — only the cached extraction results are wiped.

Run order: Video tags first (so the base dv tag exists), then DV detail layers profile / layer / CM tags on top. Quick fix-all + schedules sequence them automatically.
Tag character limits: Radarr only accepts a-z, 0-9, and - in tag labels. Stick to those when picking a custom prefix.

What this page is for

A list of every tag on the chosen instance, with how many movies or series carry each one. Use it to check what tagarr actually did — for example, after a scan you can confirm that all your FLUX releases really got the flux tag, or spot tags that aren't used anywhere and clean them up.

Find items by tag (search)

The search field at the top lets you ask questions about your library — "which movies have these tags?" — and shows the answer as a list of items, with the matching tags highlighted in blue. Type a query and the results appear above the tag table; clear the field to hide them again. Some practical examples you can paste in:

  • flux nbcu — items that have both the flux and nbcu tags (writing two tags side by side means AND)
  • flux OR thefarm — items that have at least one of the two
  • not fel or -fel — items that don't have the fel tag (use the actual tag name as it appears in your library — drop or add a prefix to match what you've configured)
  • rg-* — items that have any tag starting with rg- (the * is a wildcard at the end)
  • not bucket:dv-detail — items with no DV-detail tag at all (the bucket macro covers every tag in that group, no matter what prefix you've configured — recommended when you don't know the prefix)
  • (flux OR sic) AND not rg-old — combine with parentheses for grouped expressions

Available bucket names: resolution, codec, hdr, dv-detail, audio-codec, audio-channels, audio-flags, audio (everything audio).

Click any result row to see the file context (release group, scene name, file path) so you can spot mistagged items at a glance. The library is loaded once and cached — hit Refresh if you've changed tags in Radarr/Sonarr and want fresh data.

Pick Radarr or Sonarr first

The two buttons at the top — Radarr and Sonarr — decide what kind of instances show up below. Click Radarr and the dropdown only lists your Radarr instances; click Sonarr and you get your Sonarr instances. Everything on the page (tags, search, rename, delete) only touches the instance you have picked. If a button is greyed out, you simply don't have any instance of that kind set up yet — go to Settings → Instances to add one.

Rename one tag

Click Rename on a row to give the tag a new name. The movies or series that already had the old name keep the tag — it just shows up under the new name afterwards. Handy when you want to tidy up names without losing what was already tagged. Example: rename flux to rg-flux and all 47 movies with the tag now show rg-flux instead.

Rename many tags at once

Tick the checkbox on two or more rows, then click Batch rename. You can add the same prefix or suffix to all of them, or find-and-replace a piece of text. Practical example: tick flux, sic and thefarm, pick Add suffix, type -old, and they all become flux-old, sic-old and thefarm-old in one go. The preview table shows you exactly what each tag will end up as before you press Apply, so there are no surprises.

Delete a tag

Click Delete to remove a tag from every movie or series that has it, and remove the tag itself from the instance. Useful for cleaning up tags you no longer want. Be careful: if anything else reads the tag (a Quality Profile, a Custom Format, a third-party cleanup tool) it loses that reference — so double-check before deleting widely-used tags.

Compare two tags

Tick the checkbox on exactly two rows, then click Compare selected. You'll see three lists: items that carry both tags, items with only the first one, and items with only the second. Practical example: pick flux and web-dl to see which of your FLUX releases are actually WEB-DLs, and which ones got the wrong source tag. Useful for catching mis-tags or finding tags that overlap so much you might want to merge them.

Compare the same tag across two instances

If you have more than one instance of the same kind (for example a primary Radarr and a 4K Radarr), the dropdown next to Compare appears. Pick the other instance, tick exactly one tag in the table, and click Compare across — it looks up the same tag name on the other side and shows you which movies have it both places, only on the current one, or only on the other one. Practical example: you sync your flux tag from primary to 4K via tag-mode. Pick 4K from the dropdown, tick flux, and the result tells you exactly which FLUX movies haven't made it across yet — a 30-second sanity check on whether your sync workflow is keeping up. Items that don't have a matching ID on both sides (movies without tmdbId, series without tvdbId) are skipped and noted at the top of the result.

Tip: Click anywhere on a tag row to expand it and see the full list of items that carry the tag — with filename, scene name and release group shown side-by-side, so you can quickly spot anything that ended up tagged by mistake.

Used to preserve tokens that exist in the indexer's release title but might be missing from the qBit torrent name (which Arr reads at import time for CF scoring), and to tag the torrent in qBit so backlogs stay organized. Files on disk are untouched — the rename is metadata-only.
Rename criteria
Radarr / Sonarr
When to apply the qBit torrent rename. Idempotent — skips when the qBit name already matches what tagarr would set.
qBit season / episode tagging
Sonarr only M7
While the Grab handler has the qBit torrent open for the rename, also tag it with the season/episode it belongs to. Use the Run backlog scan button under Settings → qBittorrent to apply the same tags to existing torrents.
On Import / On Upgrade
Apply tags right after Radarr / Sonarr finishes importing a file, or re-evaluate them when an upgrade replaces an existing file (release group may have changed).
